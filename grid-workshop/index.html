Grid is the new black

Revolutionising visual design on the web

This workshop assumes you have a reasonably good grasp of CSS, and how to handle specificity and the cascade. That you could write an entire site’s layout in vanilla CSS without any frameworks.

If you are a bit more novice on the CSS front, no worries, just that the pace might be a little fast. But you can ask our facilitator, Shirlaine, for help if you get really lost.

Just to get a grasp of the audience today, how many people have used CSS grid before? In production? How many of you use Flexbox day-to-day? And what’s the most common technique you use to build your layouts? Floats? Tables?

If you ask me, I will always say context matters. That there is never a single way to achieve something, so don’t treat whatever you learn today as dogma, or the ONE WAY. Think of this as adding tools to your CSS toolbox, and it’s up to you to choose the right tool for the right situation.

Does everybody have Firefox Nightly installed? And I think you all should have received some instructions via email outlining the different ways you can set this up. At the end of the day, everything we are doing is vanilla HTML and CSS. There is nothing extra to install to get your code to work.

Nightly is recommended so you can make use of the Grid Inspector tool, and the starter kit is merely a workflow that I personally use because I’m too lazy to press Command-Shift-R hundreds of times a day. Like I said, the assumption is that this audience comprises of experienced web developers, so do whatever suits you best.

At the very least, just have an HTML file and write your styles between <style> tags in the <head>, or ideally in a separate style sheet. CodePen is definitely an option, I just wasn’t sure how stable the internet connection was going to be. Pencil and paper is a plus, you'll see.

Everybody set up? Good. Let’s begin.

Rendering engines 101

Images and explanations by Lin Clark

Anybody who worked/works in an agency? How often do you get designs that are very “print”? I believe not all agencies are the same, but very often, as the developer, I ended up with one comp, for desktop, and a bunch of people expecting the website to look exactly like the psd, down to the pixel.

Sometimes you’d get a comp for the mobile version too, and that usually made my life worse, because the mobile version’s layout isn’t even remotely related to the desktop version with regards to layout.

I truly believe this is because of a lack of understanding of how the web as a medium works. And I admit, earlier on in my career and life, I wasn’t the best communicator. I was kinda sullen, and would give my designers all varieties of dirty looks. So immature, really.

Does everyone know how a browser renders elements on the page? If so, just bear with me for a moment, I’ll try to give the abridged version. These graphics are from Lin Clark, author of Code Cartoons, and someone I really look up to in the web industry, and have personally learned a lot from.

Images and explanations by Lin Clark

HTML and CSS are human-readable (largely), but the rendering engine needs to parse the information into objects the browser can understand. At this point in the rendering process, the DOM knows about how the page is structured and all the parent/child relationships between elements. It just doesn't know HOW those elements should look like yet.
